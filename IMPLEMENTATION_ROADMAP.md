# SemiPLM 实施路线图

## 项目记忆与迭代规则

本项目以用户在对话中持续补充的业务判断为准。每次出现新的产品目标、行业口径、数据口径、交互标准或不合理设计纠偏，必须同步更新到 Markdown 文档，再继续开发，避免只在当次代码里临时修补。

- 以通用商用 PLM 能力为主线，满足光电芯片制造行业；武汉睿码功能清单作为主要参考，鼎捷 PLM 仅作为辅助参考，主流 PLM 的共性能力可作为功能边界校准。
- 完整菜单和功能域设计记录在 `MENU_ARCHITECTURE.md`；后续页面开发必须先对齐菜单边界，再实现功能，避免把多个管理对象粗糙塞进一个页面。
- 面向光电芯片制造行业，目标是可真实商用的轻量单机 PLM，不做玩具页和纯演示页。
- 单公司、单部门、单组织场景；公司为南智光电，当前部门为生产部。
- 负责人、经理、执行人、审核人等人员字段必须来自用户/角色主数据下拉，不允许随意手输。
- 产品、文档、BOM、工艺、变更、流程、基础配置等配置对象必须具备真实 CRUD；不能只做新增或只做展示。
- 表格操作区必须紧凑、横向、不换行；删除动作使用小按钮并二次确认，不使用整条红色大按钮。
- 页面文案禁止出现“演示环境”“单组织部署”“当前唯一业务部门”“队列就绪”等实现或演示口径。
- 当前阶段优先级：业务功能合理性和完整性优先于复杂工程严谨性。先做真实对象、真实 CRUD、真实流程闭环和合理页面交互；事务一致性、高并发、复杂锁、审计强化、性能分片等企业级工程治理暂缓到业务主线成熟后再补。
- 目标部署环境为 Linux + Docker；Windows 仅作为当前开发环境。代码、脚本、路径和文档必须避免绑定 Windows 盘符或 PowerShell 专属习惯。
- 日常页面和功能迭代优先提高开发效率：小改动以页面可打开、控制台无明显错误、关键交互可用为主；涉及后端接口、数据结构、部署、发布前再做后端编译、前端构建和更完整验证。

## 完整闭环标准

每个功能点必须满足以下标准，才算完成：

- 有明确业务对象和对象边界。
- 有数据库表或明确复用的数据模型。
- 有后端 API，不能只做前端静态页面。
- 支持必要的 CRUD 或业务动作。
- 有生命周期状态。
- 有角色/权限关系。
- 需要审核的对象必须接入流程模板和流程实例。
- 对象之间有真实关联，不做孤岛页面。
- 关键动作有结果反馈和可追溯记录。
- 涉及 ERP/MES/QMS 的动作必须进入集成队列。
- 页面布局符合商用后台：列表、筛选、详情、动作区清晰，不堆按钮，不把无关功能塞一个页面。
- 主数据字段必须使用受控数据源：人员从用户主数据选择，角色从角色主数据选择，产品/物料/文档/BOM/工艺从对应对象选择。
- 配置类页面必须完整支持新增、编辑、删除、状态维护和必要的删除确认。
- 表格操作列必须保持单行紧凑，不允许按钮换行撑高行高。
- 完成后必须验证：后端编译、前端构建、接口请求、浏览器页面检查。

## 阶段 1：信息架构与系统底座

目标：先让系统像商用 PLM，有合理二级菜单和可维护的底座配置。

- 重构二级菜单。
- 拆分系统管理大页面。
- 用户管理。
- 角色管理。
- 权限管理。
- 编码规则。
- 分类管理。
- 属性模板。
- 生命周期模板。
- 流程模板。
- 枚举字典。
- 权限模型：功能权限、数据权限、字段权限。

验收：
- 菜单结构与功能清单一致。
- 每个配置对象有独立页面。
- 用户、角色、流程、集成配置真实 CRUD。
- 页面无横向溢出。

范围约束：
- 只做单公司/单工厂/单组织。
- 不做多组织、多租户、多法人、多站点隔离。
- 部门字段保留在用户资料和流程分派中，不做独立组织树模块。

阶段调整：
- 集成端点配置移至阶段 7 集成中心；当前已实现的基础接口配置页面保留在集成中心，不再视为底座验收核心。
- 阶段 1 不做复杂组织树，只做用户、角色、权限、编码、分类属性、生命周期、枚举字典、流程模板。

## 阶段 2：流程平台闭环

目标：文档、BOM、变更、项目阶段门都能走统一流程。

- 流程模板。
- 流程节点。
- 流程实例。
- 流程任务。
- 我的待办。
- 审核通过/驳回。
- 转办。
- 撤回。
- 驳回至指定节点。
- 超时提醒。
- 流程日志。
- 业务对象状态联动。

验收：
- 文档提交生成流程实例。
- BOM 提交生成流程实例。
- 工作台可处理待办。
- 最后节点通过后自动发布对象并生成集成队列。
- 任意业务对象可通过配置挂载流程模板，不为每类对象硬编码独立流程。

## 阶段 3：主数据中心

目标：产品、物料、需求、文档成为完整主数据。

- 产品库。
- 产品版本。
- 产品资料包。
- 需求规格。
- 物料库。
- 原材料库。
- 文档库。
- 文档版本。
- 文档关联对象。
- 编码规则自动生成对象编号。
- 分类属性继承。

验收：
- 所有主数据可维护。
- 版本和状态可追溯。
- 产品详情能看到关联对象全景。
- 产品资料包只作为产品详情聚合视图，不做独立业务对象。

## 阶段 4：BOM 与工艺

目标：实现研发制造核心结构。

- EBOM 多层结构。
- EBOM -> PBOM 转换。
- BOM 明细维护。
- BOM 版本。
- BOM 比较。
- BOM 审核发布。
- BOM 反查。
- BOM 有效性：生效日期、失效日期、生效批量。
- 工艺路线。
- 工艺步骤。
- 工艺参数。
- 工序物料分配。
- 工装/辅料。
- 工序检验标准。
- 工艺发布。
- BOM 与工艺关联。

验收：
- BOM 可管理、可审核、可发布。
- 工艺可管理、可审核、可发布。
- 发布后进入 ERP/MES 集成队列。
- 阶段 4 只产生集成事件；发送、重试、日志和回调归阶段 7。

## 阶段 5：变更管理

目标：PR/ECR/ECO/ECN/ECA 全链路。

- 问题报告 PR。
- ECR 变更申请。
- 影响分析。
- 系统自动反查：where-used、关联文档、关联项目、关联工艺。
- ECO 变更指令。
- ECN 变更通知。
- ECA 执行动作。
- 变更生效范围。
- 下游同步。

验收：
- 变更能关联产品、BOM、文档、工艺、Lot、项目。
- 变更审批完成后驱动对象版本和集成队列。
- 变更创建临时版本，审批通过后自动升版并归档历史版本。

## 阶段 6A：项目管理

目标：NPI/APQP 阶段门和项目交付物闭环。

- 项目库。
- 项目模板。
- APQP/NPI 阶段门。
- 项目计划。
- 项目交付物。

验收：
- 项目阶段门能驱动交付物。
- 交付物审批发布后，自动完成对应任务节点并更新项目进度。

## 阶段 6B：研发质量闭环

目标：质量问题、CAPA 与变更联动，不替代 MES 生产过程采集。

- 质量问题。
- CAPA。
- 质量报告归档。
- 质量问题触发 ECR。

验收：
- 质量问题能触发变更。
- CAPA 有闭环状态。
- 不做 MES 的 CP/FT 在线采集和完整 Lot 生产过程管理。

## 阶段 7：集成中心

目标：当前只做发布对象的轻量队列、状态回写和追溯，保证 ERP/MES/QMS 接入前系统内业务闭环不被阻塞；字段映射、接口日志平台、下游回调等能力等实际对接时再扩展。

- ERP/MES/QMS 接口配置。
- 集成队列。
- 轻量同步回执。
- 失败重试。
- 对象同步状态追溯。
- 对象映射规则（暂缓到真实集成）。
- 字段级映射规则（暂缓到真实集成）。
- 手动重发。
- 下游回调。
- 集成监控看板。

验收：
- 发布对象进入队列。
- 队列可处理、失败、重试。
- 日志可追溯。
- 支持 ERP/MES/QMS 回传同步结果并更新队列状态。
